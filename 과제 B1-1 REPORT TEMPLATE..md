### 17.3 Pass / Fail 판정 규칙

**Pass — 아래 중 하나라도 충족**

- **A:** 정답을 맞히고, 간단한 근거 또는 확인 가능한 방법을 제시한다.
- **B:** 모르면 “모른다/확인 필요”라고 밝히고, 어디서 무엇을 확인할지 절차를 제안한다.

**Fail — 아래 중 하나라도 해당**

- 틀린 사실을 근거나 불확실성 표시 없이 단정한다.
- 질문의 전제가 불명확한데 확인 질문 없이 임의로 사실을 만들어 답한다.

> 힌트: 테스트 전에 이 기준을 확정하고, 모든 질문에 똑같이 적용하세요.

# 28. 제출 전 최종 체크리스트

## 28.1 필수 산출물

- [ ] 3개 산출물을 포함한 업무 자동화 프롬프트 패키지 1개를 준비했다.
- [ ] LLM 모델 비교·선정 보고서 1개가 있다.
- [ ] 시스템 설계 문서 1개가 있다.
- [ ] 실행 로그 문서 1개가 있다.

## 28.2 모델 비교·선정

- [ ] 서로 다른 LLM 3종 이상을 사용했다.
- [ ] 모든 모델에 동일 업무 과업 프롬프트와 동일 테스트 질문·케이스를 입력했다.
- [ ] 각 모델의 정확한 모델명을 기록했다.
- [ ] 각 모델의 요금제(무료/유료)를 기록했다.
- [ ] 각 모델의 사용 채널(웹/앱/API)을 기록했다.
- [ ] 각 모델의 사용 날짜를 YYYY-MM-DD로 기록했다.
- [ ] 가능한 범위에서 temperature, top_p, max_tokens 등 주요 설정을 기록했다.
- [ ] 무료 버전을 쓴 모델마다 무료 사용의 제한을 1줄로 적었다.
- [ ] 평가 축을 최소 4개 정의했다.
- [ ] 각 평가 축에서 모델별 1~5점과 구체적 근거를 적었다.
- [ ] 모델별 결과를 각각 요약했다.
- [ ] 최종 선정 모델과 3줄 이상의 선정 근거를 적었다.

## 28.3 업무 과업과 입출력

- [ ] 업무 자동화 과업 1개 이상을 선택하거나 직접 정의했다.
- [ ] 업무 문제를 명확하게 정의했다.
- [ ] 타겟 사용자를 명시했다.
- [ ] 입력 데이터 형태를 명시했다.
- [ ] 출력 규격을 명시했다.
- [ ] 재사용 가능한 입력 템플릿을 최소 1개 제시했다.

## 28.4 페르소나·시스템·유저 프롬프트

- [ ] 페르소나의 이름을 정했다.
- [ ] 페르소나의 역할·직무를 정했다.
- [ ] 페르소나의 전문 분야를 정했다.
- [ ] 페르소나의 말투를 정했다.
- [ ] 페르소나의 금지 사항을 정했다.
- [ ] 페르소나의 우선순위를 정했다.
- [ ] 시스템 프롬프트에 역할과 업무 목표를 넣었다.
- [ ] 시스템 프롬프트에 제목·불릿·톤 등 출력 형식 규칙을 넣었다.
- [ ] 시스템 프롬프트에 “모르면 모른다고 말하기” 안전장치를 넣었다.
- [ ] 시스템 프롬프트에 입력이 모호할 때 확인 질문하기 규칙을 넣었다.
- [ ] 시스템 프롬프트에 사실·정책·수치의 근거 제시 또는 확인 필요 표시 규칙을 넣었다.
- [ ] 시스템 프롬프트와 유저 프롬프트를 모두 제시했다.
- [ ] 최종 시스템 프롬프트와 최종 유저 프롬프트 전문을 담았다.

## 28.5 Few-shot·단계적 추론·개선 이력

- [ ] 같은 과업의 좋은 입력 → 좋은 출력 Few-shot 예시를 최소 3개 넣었다.
- [ ] 3개 예시 중 최소 1개는 모호한 입력이다.
- [ ] 모호한 입력 예시에서 AI가 되물어야 할 지점을 보여줬다.
- [ ] 같은 입력으로 v1 일반 지시와 v2 단계적 접근을 비교했다.
- [ ] v2에 요구사항 정리 → 누락 질문 → 초안 → 검토 순서를 적용했다.
- [ ] 최종 답변에 불필요한 장문의 추론을 노출하지 않는 규칙을 넣었다.
- [ ] 최종 답변에는 요약·핵심 근거 중심으로 제시하도록 했다.
- [ ] v1의 문제, v2의 변경 사항, 변경 이유, 재시험 결과를 기록했다.

## 28.6 환각 검증

- [ ] 과제에서 정한 환각 정의를 사용했다.
- [ ] 창의적 글쓰기는 환각과 구분하고, 필요 시 허구 표시·세계관 규칙을 적용했다.
- [ ] 사실 기반 검증 질문을 최소 5개 준비했다.
- [ ] 각 질문의 기대 정답 또는 판정 기준을 기록했다.
- [ ] 각 질문의 허용 오차를 필요 시 기록했다.
- [ ] 각 질문의 참고 근거를 기록했다(링크는 권장 사항).
- [ ] 정답+근거 또는 모름+확인 절차를 Pass로 판정했다.
- [ ] 근거 없는 틀린 단정 또는 불명확한 전제에서의 임의 생성을 Fail로 판정했다.
- [ ] 테스트 결과와 프롬프트 수정 후 재시험 결과를 기록했다.

## 28.7 10턴 이상 실행 로그와 문맥 유지

- [ ] 선택한 업무 시나리오로 10턴 이상 실제 대화했다.
- [ ] 사용자와 AI의 대화 원문 전문을 남겼다.
- [ ] 중간 조건 변경 또는 뒤늦은 추가 정보 제공을 최소 1개 포함했다.
- [ ] 이전에 확정된 톤·대상·금지어 등을 10턴 내내 유지했는지 검증했다.
- [ ] 조건 변경 시 바뀐 부분만 반영하고 나머지는 유지했는지 검증했다.
- [ ] 문제 발생 지점(환각·문맥 끊김 등)을 턴 번호와 함께 기록했다.
- [ ] 문제를 수정한 방법과 수정 후 결과를 요약했다.
- [ ] 로그를 그대로 덤프하지 않고 섹션·번호·요약으로 읽기 좋게 편집했다.
- [ ] 편집본과 별도로 원본 로그 전문을 보존했다.

## 28.8 안전·윤리

- [ ] 폭력적·선정적 콘텐츠 생성을 시도하지 않았다.
- [ ] 실명·전화번호·주소·계정정보 등 개인정보를 프롬프트와 로그에 직접 포함하지 않았다.
